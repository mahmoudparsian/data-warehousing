# 🟦 What is EDA in Data Warehousing?

* 👉 EDA is the foundation of trustworthy data warehouses


- Exploratory Data Analysis (EDA) = systematic profiling of data before modeling
- In DW: applied to Bronze → Silver transition
- Focus:
  - Data structure
  - Data quality
  - Data meaning (semantics)

👉 EDA answers: “Can we trust this data for analytics?”

---

# 🎯 Goal of EDA

- Understand data shape and distribution
- Identify data quality issues early
- Validate business meaning of attributes

👉 Goal:  
Turn raw data → trusted data foundation

---

# 🧠 Purpose of EDA (DW Perspective)

EDA ensures:

- ✔ Data is accurate
- ✔ Data is consistent
- ✔ Data is complete
- ✔ Data is business-aligned

👉 Prevents bad data entering Gold layer

---

# 🏗️ Where EDA Fits (Medallion Architecture)

- 🟫 Bronze → raw ingestion (no trust)
- ⚪ Silver → cleaning + validation
- 🟡 Gold → analytics-ready

👉 EDA is:

- Pre-Silver (profiling)
- During Silver (validation rules)

---

# 🔍 Types of EDA Analysis (VERY IMPORTANT)

## 1. Structural Analysis
- Column names, types
- Schema consistency

## 2. Completeness Analysis
- NULL counts
- Missing ratios

## 3. Uniqueness Analysis
- Duplicates
- Key validation

## 4. Distribution Analysis
- Min, max, avg
- Skew, spread

## 5. Validity Analysis
- Domain checks (e.g., country list)
- Format checks (email, date)

---

# 📊 Example SQL for Each Type

### COMPLETENESS ANALYSIS

```sql
-- =========================
-- COMPLETENESS ANALYSIS
-- =========================

SELECT 
    COUNT(*) - COUNT(email) AS missing_email
FROM table;
```

### UNIQUENESS ANALYSIS

```sql
-- =========================
-- UNIQUENESS ANALYSIS
-- =========================
SELECT 
    COUNT(*) - COUNT(DISTINCT employee_id) 
      AS duplicate_records
FROM table;
```

### DISTRIBUTION ANALYSIS

```sql
-- =========================
-- DISTRIBUTION ANALYSIS
-- =========================

SELECT 

    MIN(salary) AS min_salary,
    MAX(salary) AS max_salary,
    AVG(salary) AS avg_salary

FROM table;
```

### VALIDITY ANALYSIS

```sql
-- =========================
-- VALIDITY ANALYSIS
-- =========================

SELECT *

FROM table

WHERE country NOT IN ('USA', 'CANADA');
```

# 🧪 Semantic Analysis (Critical in DW)

EDA is not just technical:

👉 Ask business questions:

- Does salary make sense?
- Are values realistic?
- Do categories reflect real business entities?

Example:

```
text id="eda2" 
Salary = -100 ❌ 
Country = 'Mars' ❌ 
```

🧠 Interpretation

* Salary = `-100`
    * Salaries should not be negative
    * 👉 Indicates:
        * data entry error
        * parsing issue
        * corrupted data
        
* Country = `'Mars'`
    * Not a valid business value
    * 👉 Indicates:
        * invalid category
        * bad source data
        * lack of validation rules

---

# ⚠️ Common Data Issues Found

- Missing values (NULLs)
- Duplicate rows
- Invalid dates (e.g., Feb 30)
- Outliers (salary = 1,000,000)
- Inconsistent categories

👉 These become Silver cleaning rules

---

# 🧭 EDA → Silver Transformation Mapping

| EDA Finding | Silver Action |
|------------|--------------|
| Missing values | Impute / reject |
| Duplicates | Deduplicate |
| Invalid values | Filter |
| Bad formats | Parse / standardize |

👉 EDA drives data cleaning logic

# 📈 Advanced EDA (DW Students Level)

- Correlation checks (salary vs age)
- Segment analysis (by department)
- Time trends (joined_date)
- Outlier detection

Example:

```
sql id="eda3" 
SELECT department, 
       AVG(salary) 
FROM employees 
GROUP BY department; 
```

# 🚀 Best Practices

- Start simple → then go deeper
- Always profile each column
- Validate against business rules
- Document findings (VERY IMPORTANT)
- Never skip EDA
